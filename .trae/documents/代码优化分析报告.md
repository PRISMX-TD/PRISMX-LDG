经过全面检查，我发现了以下几个关键的可优化点，按重要性排序：

### 1. ⚡ 严重性能隐患：交易查询逻辑
- **问题**: 后端 `getTransactions` 方法目前是**先从数据库拉取该用户所有交易**，然后在内存中进行过滤和分页。
- **后果**: 随着交易记录增加（例如超过 1000 条），API 响应会越来越慢，最终可能导致服务器内存溢出。
- **优化建议**: 将过滤条件（日期、分类、搜索）和分页逻辑（Limit/Offset）直接下推到数据库 SQL 查询中执行。

### 2. 🐢 数据库查询效率 (N+1 问题)
- **问题**: 在获取交易列表时，对于每一笔“转账”交易，后端都会额外发起一次数据库查询来获取“对方钱包”信息。
- **后果**: 如果一页显示 50 条转账，就会产生 51 次数据库查询，极大拖慢速度。
- **优化建议**: 在主查询中直接使用 `LEFT JOIN` 关联对方钱包表，一次查询获取所有数据。

### 3. 🔍 数据库索引缺失
- **问题**: `transactions` 表的查询非常频繁，但 `userId`、`date`、`categoryId` 等常用过滤字段没有索引。
- **优化建议**: 为这些高频查询字段添加数据库索引，显著提升查询速度。

### 4. 📱 前端数据加载策略
- **问题**: 仪表盘 (`Dashboard`) 页面目前会请求**所有**历史交易数据，只为了显示最近的几条。
- **优化建议**: 仪表盘的交易列表请求应明确加上 `limit=10` 参数，仅获取最新数据。

### 5. 🌐 外部 API 稳定性
- **问题**: 汇率接口依赖外部免费 API，且没有任何缓存机制。
- **后果**: 每次用户刷新页面都会请求外部接口，容易触发限流或因外部服务波动导致页面报错。
- **优化建议**: 在后端添加简单的内存缓存（如缓存 1 小时），减少外部请求频率，提升稳定性。

### 6. 🧹 代码健壮性
- **问题**: 部分类型定义（如 `any`）过于宽松，且 API 错误处理比较基础。
- **优化建议**: 收紧 TypeScript 类型定义，完善错误边界处理。
